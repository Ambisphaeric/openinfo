## 2. Dependency rules (the ones that prevent the refactor spiral)

1. `apps/*` depend on `shared/contracts` and **nothing else in the repo**. Client never imports engine;
   workbench never imports either — they meet only at the API.
2. Only `engine/store` opens a database handle. `route/` asks store to move a session; `index/` asks store to
   write an entity. Workspace isolation is enforced by one module, so it cannot be reached around.
3. `templates/` contains no code. If a template "needs" code, a block type is missing — add it to the engine
   and keep the template a document.
4. `spikes/` is not a workspace package and is banned from imports (CI-enforced). Spikes graduate by being
   **rewritten** under `apps/` against contracts — never copy-pasted.
5. New feature → find its row in §3. If it has no row, it gets a design note in ARCHITECTURE.md *before* code.

## 4. Spikes — what they are and how they bend the plan

A **spike** is a throwaway proof that answers exactly one question, in hours, in `spikes/`, with none of the
production rules applied (hardcode anything, skip types, no tests). It is deleted or graduated-by-rewrite;
its output is an *answer*, recorded as a paragraph in the relevant phase doc.

|---|---|---|

### The honest recalibration you asked for

You're right: the first slice is **hours, not days**. The 6-month figure in IMPLEMENTATION.md conflates two
different clocks, and it's worth splitting them permanently:

- **Construction time** — writing the code. With agent-driven development this compresses ~10×: the Phase 1
  seam is a day-or-two build; the Phase 2 HUD vertical is a week-scale build, not six.
- **Convergence time** — the part that does not compress: dogfooding against *real* workdays. Extraction
  quality, router attribution, register detection, drift thresholds — these are tuned against lived meetings,
  and a week of calendar time contains only a week of meetings. Exit criteria like "attend a real meeting,
  trust the ledger" are experiential, not constructional.

Revised expectation per phase: **demo-able in days, trustworthy in weeks.** The phase *order* and *exit
criteria* in IMPLEMENTATION.md stand unchanged — what changes is that construction stops being the bottleneck
after week one, and the critical path becomes fixture data + dogfood cycles. That is why `tools/fixtures`
(record a meeting once, replay it deterministically) is in the tree at P0-priority: it converts convergence
time into regression tests and is the single highest-leverage tool in the repo.

Practical cadence this implies: run spikes `seam-echo`, `glass-capture`, and `extract-quality` in the first
days (they are cheap and they de-risk the two riskiest phases before any production code hardens); build
Phases 0–2 as fast as agents allow; then let the calendar do what only it can — accumulate real sessions —
while construction runs ahead on Phases 3–4 behind flags.

### The pill visibility fix wave (⌘\ anchor · fill · reopen dedupe · anchor-path e2e — branch fix/pill-visibility)

The owner's 0.0.14 live-QA could not find the pill. Three root causes, fixed exactly; the anchor-path e2e is the policy lesson made durable.

- `apps/client/src/main/config.ts` — `DEFAULTS.surfaceId` repointed `surf-openinfo-hud` → **`surf-openinfo-pill`**: the ⌘\ accelerator + tray Show/Hide act on the anchor window `createHudWindow` opens from this default, so the pill (the MVP glance face) is now what the user summons instead of the old hud. Nothing persists `surfaceId` (client.json/window-state/apps-state), so no migration; `OPENINFO_SURFACE` still overrides for dev. `createSurfaceWindow` resolves the pill's full spec (hud chrome + focusable), the `PillController` is its sole height authority, and `showHud`/`hideHud` only toggle visibility — never fighting the panel extents. Pinned in `config.test.ts`.
- `apps/client/src/surfaces/hud/pill-styles.ts` + `hud/dev-entry.ts` — **the microsquare fix.** `.pill-app` gained the `.hud` fluid-fill invariant (`width:100%;max-width:660px;min-width:0`) and `.pill-bar` the `.hud` glass-card idiom (`--s-glass` fill, backdrop blur, `--s-line` border, 15px radius, floating shadow) so the collapsed pill reads as one solid card, not naked floating buttons. - `apps/client/src/main/shell.ts` — `reopenPersistedApps` already deduped the anchor (`if (surfaceId === cfg.surfaceId) continue`, keyed to the exact id `createHudWindow` builds the anchor from), so repointing the default makes a QA-era persisted `surf-openinfo-pill` entry skip automatically instead of opening a DUPLICATE second pill; comment made explicit. - `apps/client/scripts/pill-e2e.mjs` (`test:e2e:pill`) — **SCENE 0, the anchor path.** POLICY (script header): a driven e2e enters through the user's ENTRY POINTS, not past them — scenes 1-6 drove the pill window directly and stayed green through the anchor gap. Full driven set green 9/9.

### The Ask-face resolve race fix (retry · true disabled reasons · race-path e2e — branch fix/ask-face-resolve-retry)

The packaged shell creates the pill window BEFORE `ensureEngine()` spawns the bundled engine, and the Ask-face resolve was a one-shot — losing that race left the Ask button (the door to the whole chat box) dead on every packaged cold boot, with a tooltip that lied about why. Fixed with the boot controller's own posture.

- `apps/client/src/surfaces/hud/dev-entry.ts` — the Ask-face resolve now runs through `createAskResolveController`: the boot controller's capped backoff (`backoffMs`) retries `resolvePillAskSurface` until it succeeds (⇒ `setAskAvailable(true)` + rerender, exactly as before), stopping ONLY on the new typed `NoChatFaceError` — thrown when GET /bundles ANSWERED and the data says this pill's bundle has no chat face (a data answer; retrying is pointless). The transient/terminal distinction is a TYPE, never a message match. Every transient failure `console.error`s its attempt (the trace the old one-shot `.catch` never left); the terminal reason paints via `pillSources.chatError`. Asserted headless in `ask-resolve.test.ts` (manual-scheduler idiom from boot.test.ts): fails-N-then-succeeds enables Ask, the ladder caps at 8s forever, no-chat-face settles once with zero retries.
- `apps/client/src/surfaces/hud/pill-layout.ts` — `askDisabledTitle`: the disabled Ask affordance discloses the TRUE current reason — `Ask — catching up, chat will be ready in a moment` while the retry loop is still working; `Ask — this app has no chat face` only after the terminal data answer (the old static title claimed no-chat-face during the mere engine-spawn race). The honest Ask panel's resolving copy is human now ("Catching up — chat will be ready in a moment."). Both disabled reasons covered by the interaction lint + layout tests (the catching-up state must never render the no-chat-face wording).
- `apps/client/scripts/ask-race-e2e.mjs` (`test:e2e:askrace`) — the tenth driven e2e, the ENGINE-LATE race path no prior e2e entered (they all started the fake engine before the window — why this bug shipped green): anchor window at t=0 (Scene-0 entry-point policy), nothing listening; honest boot chip asserted while down; engine listens ~2.5s later; the regression lock — Ask BECOMES enabled once the engine is healthy (verified to FAIL against the pre-fix one-shot) — then the user's Ask click mounts the chat panel + text input at the real 432px extent and typed keystrokes land.
