openinfo — surface renderings

An always-on companion split in two: a thin client that captures and renders, and an engine — local by default, or any HTTP address — that distills, routes, and drafts. Modes are the unit of configuration. When hardware can't keep up, work queues instead of dropping.

02

Mode configuration

source · distill — the honest hardware contract

A mode is the unit of configuration, and it's allowed to exceed your hardware. The envelope check does the math — measured tokens per second against what the mode demands — and when you're short, you pick the fallback: queue raw capture for later processing (then delete), degrade cadence, or drop. The engine itself is just another setting: this machine, or an IP and a port.

08

Code organization — planned before code exists

where loom fits · where the client/engine seam is

The seam that makes the remote backend possible is the same seam that prevents the old failure: the client never touches a database. It captures and renders; everything else happens behind the engine's HTTP/WS API, whether that engine is on localhost or another machine. Workspaces live where they always should have — in the store, one file each.

openinfo/ ├─ apps/client/ thin client · Electron (glass shell) │ ├─ capture/ mic, screen Δ-gate, SystemAudioDump, aec — from glass │ ├─ surfaces/ hud, modes, timeline, inbox, analytics, explore │ └─ engine-link/ http/ws client · offline spool when engine unreachable ├─ apps/engine/ daemon · localhost by default, any host by config │ ├─ api/ HTTP+WS: /modes /sessions /queue /recall /events │ ├─ bus/ ← loom packages/bus │ ├─ workflow/ ← loom packages/recipe — DAG executor runs modes │ ├─ distill/ rolling merge, screenshot distiller, prompts │ ├─ route/ context-switch detector, workspace attribution, identity (your name/aliases) │ ├─ store/ ← loom packages/store + sqlite-vec — one DB file per workspace │ ├─ queue/ deferred backlog, drain scheduler, ETA math, raw-file GC │ ├─ overlay/ tone, rules/lenses, roles, ontology │ └─ models/ provider registry, measured tok/s benchmarks └─ shared/contracts/ typed API + event schemas — nothing crosses the seam untyped
Why the seam is load-bearing
“Backend as a configuration option” means the client/engine boundary must exist on day one — you can't retrofit it. It's also the isolation fix: workspace DBs are an engine-internal concern, so no UI code can ever reach around them the way the old fork's column filters were reached around.
Loom's fit, settled: three packages transplant into the engine — bus (events), recipe (the DAG executor that runs compiled modes), and store (with its sqlite-vec adapter powering Related recall). Loom's empty engine/graph/cli stubs stay behind.
Glass's fit: the client keeps glass's capture plumbing, content-protection windowing, and hide-from-screenshare. Its Firebase layer, shared DB, and prompt presets don't cross over.
Rabbithole's fit: the explore canvas follows its serving pattern — a self-contained HTML surface delivered by the engine, split by responsibility (shell / styles / client chunks). The engine does the thinking; the canvas only renders and branches.
What changed in v0.2: telemetry moved off the HUD into Modes; the HUD gained rewind (1m/5m/30m), away-mentions, and a queue-consequence chip. New surfaces: Mode configuration with the hardware-envelope contract and remote-engine backend, Backlog analytics with drain projection, and the rabbithole-style Explore canvas. The code map fixes the client/engine seam and settles where loom's salvageable packages land. All names, numbers, and metrics shown are mock content.
